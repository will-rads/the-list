## Deliverable 2 — `web/admin.html` (founder ops, deliberately plain)
Zero design budget: system-ish styling is fine, just dark + Jakarta. One file, supabase-js + a
little vanilla JS (or babel React if genuinely faster — your call, smallest wins).
- Login: email OTP (signInWithOtp → verifyOtp). After login check own profiles row; if role !==
  'founder' show "Founders only" and stop.
- Sections (each a plain table + buttons, refreshed after every action):
  1. Applications queue: profiles where status='pending' (id, full_name, ig_handle, phone,
     created_at) → buttons Approve / Reject → rpc approve_member(p_member) / reject_member(p_member).
  2. Invite codes: list invite_codes (code, used_by, used_at) + button "Generate 10" → rpc
     create_invite_codes(p_count := 10).
  3. Create venue: inputs owner-user-uuid, name, area → rpc create_venue(p_owner, p_name, p_area).
     Helper note under it: "owner must have logged in once (profiles row exists)".
  4. Stories: stories joined applications+events (member, event, verdict, score, due_at) → buttons
     Verify / Reject (prompt for reason) → rpc override_story(p_story, p_verdict, p_reason).
  5. Bookings: bookings joined events (title, bundle_price, cut_amount, invoice_status) → button
     cycling invoice_status pending→invoiced→paid via direct update... bookings has no founder
     UPDATE policy, so skip the button: read-only table + note "invoice status: edit in Supabase
     for now". Do not add policies.
- All server writes go through the RPCs listed; do not add new SQL.
